# 專案三：智能客服與社群監測中心 - 完整實作指南

## 專案概述

**專案名稱**：智能客服與社群監測中心（Intelligent Customer Service & Monitoring Hub）
**難度等級**：⭐⭐⭐⭐⭐（高階）
**預估時間**：20小時
**核心技術**：LINE Bot / Telegram Bot、多輪對話記憶、情緒分析、輿情監測、自動報表
**AI 模型**：gpt-4o-mini
**更新日期**：2026年2月

### 學習目標

1. 建立具備對話記憶的 AI 智能客服系統
2. 實作多來源輿情監測與情緒分析
3. 建立自動化數據儀表板
4. 掌握 Supabase 雲端資料庫操作
5. 所有 AI 輸出使用 JSON 結構化格式

### 為什麼選擇 LINE Bot + Telegram Bot？

| 比較項目 | Meta/Facebook API | LINE Bot + Telegram Bot |
|---------|-------------------|------------------------|
| 申請時間 | 企業驗證需數週 | LINE：專案一已設定完成；Telegram：5分鐘 |
| 費用 | 免費但門檻高 | 皆免費 |
| Webhook | 需要公開 HTTPS URL | LINE：同上；Telegram：支援 Polling 模式 |
| 教室適用性 | 30人同時申請不現實 | 每位學員都可獨立完成 |
| 台灣市場 | Facebook 觸及率下降中 | LINE 用戶 2,100 萬，市場主流 |

### 為什麼選擇 Supabase 而非本地 PostgreSQL？

| 比較項目 | 本地 PostgreSQL | Supabase |
|---------|----------------|----------|
| 安裝難度 | Windows/Mac 各不同，30人教室是災難 | 零安裝，瀏覽器操作 |
| 連線設定 | 防火牆、port、權限問題 | 自動產生連線字串 |
| 免費額度 | N/A | 500MB 資料庫、2GB 頻寬、50,000 列 |
| REST API | 需自己建 | 內建，自動產生 |
| 管理介面 | 需裝 pgAdmin | 內建 Table Editor |

#### 5.3 解析 LINE 訊息

LINE Webhook 的資料結構與其他平台不同，需要解析：

1. 添加「Code」節點，命名為 `解析LINE訊息`
2. 程式碼：

```javascript
// 解析 LINE Webhook 事件
const body = $input.first().json.body;
const events = body.events;

// 只處理文字訊息事件
if (!events || events.length === 0) {
  return [{ json: { skip: true } }];
}

const event = events[0];

// 過濾非文字訊息
if (event.type !== 'message' || event.message.type !== 'text') {
  return [{ json: { skip: true } }];
}

return [{
  json: {
    skip: false,
    user_id: event.source.userId,
    reply_token: event.replyToken,
    message_text: event.message.text,
    message_id: event.message.id,
    timestamp: event.timestamp,
    platform: 'line'
  }
}];
```

【截圖位置：Code 節點完成設定】

#### 5.4 過濾非文字訊息

1. 添加「IF」節點，命名為 `是否為文字訊息`
2. 條件：`{{ $json.skip }}` → `is equal to` → `false`
3. True 分支繼續流程；False 分支不做任何事

### Step 6：查詢 Supabase 對話記憶

#### 6.1 取得使用者顯示名稱

添加「HTTP Request」節點，命名為 `取得LINE用戶資料`：

- **Method**：`GET`
- **URL**：`https://api.line.me/v2/bot/profile/{{ $('解析LINE訊息').item.json.user_id }}`
- **Authentication**：`Generic Credential Type` → `Header Auth`
  - **Name**：`Authorization`
  - **Value**：`Bearer 你的_CHANNEL_ACCESS_TOKEN`

> 💡 這步是為了取得使用者的顯示名稱，讓客服記錄更有意義。

#### 6.2 查詢歷史對話

添加「Supabase」節點，命名為 `查詢對話記憶`：

1. **Resource**：`Row`
2. **Operation**：`Get Many`
3. **Table Name**：`conversation_memory`
4. **Filters**：
   - `user_id` → `equals` → `{{ $('解析LINE訊息').item.json.user_id }}`
   - `platform` → `equals` → `line`
5. **Order By**：`created_at` → `Descending`
6. **Limit**：`5`

【截圖位置：Supabase 節點查詢設定】

> 💡 **為什麼只取最近 5 筆？**
> - gpt-4o-mini 的 context window 雖然有 128K tokens，但精簡的上下文能產生更精準的回覆
> - 5 筆對話已足夠理解當前對話脈絡
> - 節省 API token 成本

### Step 7：AI Agent 處理（JSON 結構化輸出）

#### 7.1 建立分類與回覆的 AI Agent

添加「AI Agent」節點（或 OpenAI 節點），命名為 `AI客服Agent`：

- **Model**：`gpt-4o-mini`
- **System Prompt**：

```
你是一位專業的客戶服務專員，負責回覆客戶私訊。

## 你的職責
1. 分析客戶訊息的意圖和情緒
2. 根據對話歷史提供連貫的回覆
3. 判斷是否需要轉交人工處理

## 問題分類定義
- product_inquiry：詢問產品功能、價格、規格、購買方式
- complaint：抱怨、不滿、退換貨、品質問題
- partnership：商業合作、廠商洽談、異業聯盟
- other：閒聊、感謝、其他無法歸類的問題

## 回覆原則
- 語氣友善專業，使用繁體中文
- 產品詢問：提供清楚的產品資訊
- 客訴問題：先表達歉意和理解，提供解決方案，並標記需要人工介入
- 合作邀約：感謝對方興趣，提供聯絡資訊
- 其他問題：友善回應

## 重要：你必須嚴格以 JSON 格式回覆，不可包含任何其他文字。
```

- **User Prompt**：

```
【對話歷史】（最近 5 則，由舊到新）：
{{ $('查詢對話記憶').all().length > 0 
   ? $('查詢對話記憶').all().map(item => 
       '客戶：' + (item.json.message || '') + '\n客服：' + (item.json.response || '')
     ).join('\n---\n') 
   : '（無歷史對話，這是新客戶的第一則訊息）' 
}}

【新訊息】
客戶：{{ $('解析LINE訊息').item.json.message_text }}

請分析並回覆。你的回覆必須是且僅是以下 JSON 格式，不可包含 markdown 語法或其他文字：

{
  "intent": "product_inquiry 或 complaint 或 partnership 或 other",
  "sentiment": "positive 或 negative 或 neutral",
  "need_human": true 或 false,
  "reply": "你的回覆內容（繁體中文）",
  "reason": "簡短說明分類理由"
}
```

【截圖位置：AI Agent 完整 Prompt 設定】

> ⚠️ **為什麼必須用 JSON 輸出？**
> 過去的做法是用正則表達式（regex）解析 AI 回覆中的「類型：xxx」這種格式，但這極度不可靠：
> - `\w+` 無法匹配底線分隔的英文（如 `product_inquiry`）在某些 regex 引擎中會有問題
> - AI 可能多加空格、換行、emoji，導致 regex 完全失效
> - JSON 格式可以用 `JSON.parse()` 一次解析，100% 可靠

#### 7.2 解析 AI 的 JSON 回覆

添加「Code」節點，命名為 `解析AI回覆`：

```javascript
// 取得 AI 回覆的原始文字
const aiOutput = $input.first().json.text || $input.first().json.content || $input.first().json.message?.content || '';

// 嘗試從回覆中提取 JSON
// 有時 AI 會在 JSON 前後加上 ```json ... ``` 標記
let jsonStr = aiOutput;

// 移除可能的 markdown code block 標記
const codeBlockMatch = jsonStr.match(/```(?:json)?\s*([\s\S]*?)```/);
if (codeBlockMatch) {
  jsonStr = codeBlockMatch[1];
}

// 嘗試找到 JSON 物件
const jsonMatch = jsonStr.match(/\{[\s\S]*\}/);
if (jsonMatch) {
  jsonStr = jsonMatch[0];
}

let parsed;
try {
  parsed = JSON.parse(jsonStr);
} catch (e) {
  // JSON 解析失敗時的降級處理
  parsed = {
    intent: 'other',
    sentiment: 'neutral',
    need_human: false,
    reply: '感謝您的訊息！我們已收到，將盡快為您處理。如有急需，請撥打客服專線。',
    reason: 'AI 回覆解析失敗，使用預設回覆'
  };
}

// 驗證必要欄位存在
const validIntents = ['product_inquiry', 'complaint', 'partnership', 'other'];
const validSentiments = ['positive', 'negative', 'neutral'];

return [{
  json: {
    intent: validIntents.includes(parsed.intent) ? parsed.intent : 'other',
    sentiment: validSentiments.includes(parsed.sentiment) ? parsed.sentiment : 'neutral',
    need_human: typeof parsed.need_human === 'boolean' ? parsed.need_human : false,
    reply: parsed.reply || '感謝您的訊息，我們會盡快回覆！',
    reason: parsed.reason || ''
  }
}];
```

- **URL**：`{{ $credentials.supabaseUrl }}/rest/v1/conversation_memory?select=intent,sentiment,need_human&created_at=gte.{{ $now.format('yyyy-MM-dd') }}T00:00:00`
- **Headers**：
  - `apikey`：你的 Supabase anon key
  - `Authorization`：`Bearer 你的_SERVICE_ROLE_KEY`

> 💡 **替代方案**：也可以用 Supabase 的 `Get Many` 節點搭配 Filter，然後用 Code 節點做彙總計算。

#### 19.2 用 Code 節點統計

添加「Code」節點，命名為 `計算客服統計`：

```javascript
const rows = $input.all().map(item => item.json);
const today = new Date().toISOString().split('T')[0];

const stats = {
  date: today,
  total_conversations: rows.length,
  complaints: rows.filter(r => r.intent === 'complaint').length,
  product_inquiries: rows.filter(r => r.intent === 'product_inquiry').length,
  partnerships: rows.filter(r => r.intent === 'partnership').length,
  other: rows.filter(r => r.intent === 'other').length,
  positive_count: rows.filter(r => r.sentiment === 'positive').length,
  negative_count: rows.filter(r => r.sentiment === 'negative').length,
  neutral_count: rows.filter(r => r.sentiment === 'neutral').length,
  human_escalation: rows.filter(r => r.need_human === true).length,
  positive_rate: rows.length > 0 
    ? Math.round(rows.filter(r => r.sentiment === 'positive').length / rows.length * 100) 
    : 0
};

return [{ json: stats }];
```

## 常見問題排除

### LINE Webhook 無法接收

1. **確認 n8n 的 Webhook URL 是否正確**：Production URL 而非 Test URL
2. **確認 LINE Developers Console 的 Webhook URL 已更新**
3. **確認 Webhook 已啟用**：LINE Console → Messaging API → Use webhook → 開啟
4. **確認 Auto-reply 已關閉**：LINE Official Account Manager → 回應設定 → 自動回應訊息 → 關閉
5. **查看 n8n Execution Log**：是否有收到請求但處理失敗

### Telegram Bot 無回應

1. **確認 Bot Token 正確**：與 BotFather 給的一致
2. **確認 Telegram Trigger 使用 Polling 模式**（非 Webhook）
3. **確認已對 Bot 發送 `/start`**：首次使用需先啟動
4. **重啟 n8n Workflow**：停用再啟用 Trigger

### Supabase 連線失敗

1. **確認 Project URL 格式**：`https://xxxxx.supabase.co`（有 https，沒有結尾斜線）
2. **確認使用 service_role key**：不是 anon key
3. **確認表格名稱完全正確**：區分大小寫
4. **在 Supabase 網頁確認表格存在**：Table Editor 可以看到

### AI 回覆非 JSON 格式

1. **強化 System Prompt**：加上「你必須且只能回覆 JSON，不可包含任何其他文字」
2. **檢查 Code 節點的降級處理**：確認 catch 區塊提供了合理的預設值
3. **測試不同的訊息類型**：確保各種情境都能正確解析
4. **檢查 AI 模型設定**：確認使用 `gpt-4o-mini`，較新的模型對 JSON 指令的遵循度更高

### NewsAPI 返回錯誤

1. **HTTP 401**：API Key 無效或過期，重新申請
2. **HTTP 429**：超過每日 100 次限制，降低觸發頻率
3. **空結果**：嘗試更廣泛的關鍵字，或使用英文關鍵字
4. **注意**：免費方案無法取得即時新聞，有 24 小時延遲

### Google Sheets 寫入失敗

1. **確認 OAuth2 授權有效**：重新授權一次
2. **確認工作表名稱正確**：「客服統計」而非「Sheet1」
3. **確認欄位順序對應正確**：n8n 的欄位名稱必須與 Sheets 標題列一致

### 技術決策記錄

| 決策 | 原方案 | 新方案 | 原因 |
|------|--------|--------|------|
| 聊天平台 | Meta/Facebook API | LINE Bot + Telegram Bot | Meta 需企業驗證，30人教室無法使用 |
| 資料庫 | 本地 PostgreSQL | Supabase（雲端 PostgreSQL） | 零安裝，免費 500MB，內建 REST API |
| 輿情資料源 | Facebook Graph API Search | RSS + NewsAPI + Google Alerts | Graph API Search 自 2014 年已棄用 |
| AI 輸出格式 | 正則表達式解析文字 | JSON 結構化輸出 + JSON.parse() | Regex 不可靠，JSON 100% 可解析 |
| 公開網址 | ngrok | LINE 原生 Webhook / Telegram Polling | ngrok 免費版有插頁廣告，會中斷 Webhook |
| AI 模型 | 未指定 | gpt-4o-mini | 性價比最高，JSON 遵循度高 |
