# 案例拆解式自動化教學法 — 補充教學計畫

> **適用對象**：非資訊背景學員（社群行銷、品牌經營相關）
> **核心工具**：Antigravity IDE（Google 的 Agentic 開發平台）
> **設計理念**：從「看得懂的案例」倒推「為什麼能做到」，再到「自己動手做」
> **撰寫日期**：2026 年 3 月

---

## 一、問題診斷：學員卡在哪裡？

根據教學觀察，學員遇到的障礙可以分為三個層次：

| 層次 | 具體問題 | 根本原因 |
|------|---------|---------|
| **認知層** | 不知道「程式能做到什麼」 | 缺乏對電腦能力邊界的基本認知 |
| **需求層** | 不確定「我到底想自動化什麼」 | 無法將日常工作拆解為可自動化的步驟 |
| **對話層** | 與 AI 對話時「看不懂 code、不知道怎麼要求」 | 缺乏描述「輸入→處理→輸出」的思維框架 |

**關鍵洞察**：動手能力強的同學之所以成功，不是因為他們懂程式，而是因為他們會「不斷嘗試 + 修正」。我們的目標是讓所有同學都具備「知道可以試什麼」的能力。

---

## 二、教學設計哲學：「倒過來教」

### 傳統方式（由下而上）❌
```
學概念 → 學工具 → 學語法 → 試著組合 → 希望做出東西
```
**問題**：學員在第二步就失去動力，因為不知道學這些「要幹嘛」。

### 案例拆解法（由上而下）✅
```
看到成品 → 「哇，這好實用！」→ 拆解它怎麼做到的 → 發現需要什麼知識 → 帶著目的去學
```
**優勢**：學員帶著「我要做出這個東西」的動機去學每一個知識點。

---

## 三、五大拆解案例與對應知識模組

每個案例都設計為一堂「拆解課」（約 2-3 小時），結構如下：

```
[15 分鐘] 案例展示 — 「看，這就是成品」
[30 分鐘] 反向拆解 — 「它是怎麼做到的？」（學員分組討論）
[30 分鐘] 概念講解 — 「這背後的原理是什麼」
[60 分鐘] 動手簡化版 — 「我們來做一個最小版本」
[15 分鐘] 延伸思考 — 「你的工作裡，哪裡可以這樣用？」
```

### 案例一：「自動晨報機器人」— 每天早上自動彙整產業新聞發到 LINE 群組

#### 展示成品
每天早上 8:00，LINE 群組自動收到一則訊息：
> 📰 今日 AI 行銷快報（3/25）
> 1. Meta 推出新廣告自動化工具... [連結]
> 2. TikTok 演算法更新影響自然觸及... [連結]
> 3. Google 推出 AI 生成購物廣告功能... [連結]
> 💡 AI 觀點：本週品牌應關注 TikTok 觸及下降問題...

#### 學員分組討論（引導問題）
- 「新聞從哪裡來的？」→ 引出 **RSS / 網頁抓取** 概念
- 「誰讀了這些新聞然後寫摘要？」→ 引出 **AI（LLM）API 呼叫** 概念
- 「怎麼每天早上 8 點自動跑？」→ 引出 **排程 / Cron** 概念
- 「怎麼發到 LINE？」→ 引出 **Webhook / API 串接** 概念

#### 對應知識模組

| 概念 | 生活化比喻 | 技術名稱 | 在 Antigravity 中怎麼做 |
|------|-----------|---------|----------------------|
| 資料來源 | 「訂閱報紙」 | RSS Feed | Skill 裡寫：「去抓這幾個網站的最新文章」 |
| AI 處理 | 「請秘書讀完幫你寫摘要」 | LLM API Call | Skill 裡寫：「把文章丟給 AI，請它用繁中寫 50 字摘要」 |
| 定時執行 | 「設鬧鐘」 | Scheduled Task / Cron | Antigravity 的排程功能 |
| 發送通知 | 「傳訊息給朋友」 | LINE Notify / Webhook | MCP 或 HTTP Request |

#### 最小實作版
用 Antigravity IDE 建立一個 Skill：
1. 手動觸發（先不排程）
2. 從 1 個 RSS 來源抓 3 篇文章
3. 請 AI 寫摘要
4. 印出結果（先不發 LINE）

### 案例二：「一鍵多平台發文助手」— 寫一次貼文，自動轉換成 FB/IG/LinkedIn 格式

#### 展示成品
在 Antigravity IDE 輸入：
> 「幫我把這篇新品上市的貼文，分別轉成 Facebook、Instagram 和 LinkedIn 的格式」

AI 自動：
- FB 版：較長文案 + emoji + hashtag
- IG 版：短文案 + 30 個 hashtag + 建議配圖風格
- LinkedIn 版：專業口吻 + 數據引用 + CTA

#### 學員分組討論（引導問題）
- 「AI 怎麼知道三個平台的風格不一樣？」→ 引出 **Prompt Engineering** 概念
- 「為什麼 AI 能同時產出三個版本，而且格式還很整齊？」→ 引出 **結構化輸出（JSON）** 概念
- 「如果我想讓它記住我的品牌語氣？」→ 引出 **System Prompt / 上下文記憶** 概念
- 「有辦法讓它自動發佈嗎？」→ 引出 **API 權限與操作** 概念

#### 對應知識模組

| 概念 | 生活化比喻 | 技術名稱 |
|------|-----------|---------|
| 給 AI 指令 | 「跟實習生交代工作」 | Prompt Engineering |
| 格式化輸出 | 「填固定格式的表格」 | JSON 結構化輸出 |
| 品牌設定 | 「員工訓練手冊」 | System Prompt |
| 自動發佈 | 「助理幫你貼文」 | Social Media API |

#### 最小實作版
1. 在 Antigravity 寫一個 Skill 定義品牌調性
2. 輸入一段素材
3. AI 自動產出三平台文案
4. 輸出為結構化格式

### 案例三：「客戶私訊自動分流」— 粉專私訊自動判斷類型並回應

#### 展示成品
客戶在 LINE/Telegram 傳訊息給品牌帳號：
- 「你們營業時間是幾點？」→ 自動回覆營業資訊
- 「我的訂單 #12345 到哪了？」→ 自動查詢並回覆物流狀態
- 「你們產品害我過敏！」→ 標記為「緊急客訴」，立刻通知真人客服
- 「想合作業配」→ 自動回覆合作方案 PDF

#### 學員分組討論（引導問題）
- 「機器人怎麼『看得懂』客戶在問什麼？」→ 引出 **AI 分類 / 意圖辨識** 概念
- 「它怎麼決定該自動回還是轉真人？」→ 引出 **條件判斷 / If-Else 邏輯** 概念
- 「訂單資料從哪裡來？」→ 引出 **資料庫查詢 / 外部系統串接** 概念
- 「怎麼做到 24 小時都能回？」→ 引出 **Webhook（即時監聽）** 概念

#### 對應知識模組

| 概念 | 生活化比喻 | 技術名稱 |
|------|-----------|---------|
| AI 判斷意圖 | 「櫃檯人員聽懂客人要什麼」 | Intent Classification |
| 條件分流 | 「客服電話按 1 轉售後、按 2 轉業務」 | If-Else / Switch |
| 查資料 | 「翻出客戶的訂單記錄」 | Database Query / API Call |
| 24 小時待命 | 「值班電話永遠有人接」 | Webhook（即時接收）|

#### 最小實作版
1. 用 Antigravity 建立一個接收訊息的 Webhook
2. AI 判斷訊息屬於哪一類
3. 根據類別回覆不同內容
4. 「客訴」類自動發通知給管理員

### 案例四：「競品社群監測儀表板」— 自動追蹤競品動態並生成週報

#### 展示成品
每週一早上收到一份報告：
> 📊 上週競品動態摘要
> - 競品 A：發了 5 篇貼文，平均互動率 3.2%，主推「春季折扣」
> - 競品 B：新開設 Threads 帳號，首篇貼文 500 讚
> - 競品 C：被網友負評「出貨太慢」，相關討論 23 則
> 💡 建議：本週可針對「快速出貨」做差異化宣傳

#### 學員分組討論（引導問題）
- 「資料從哪裡抓的？不會要我手動去看吧？」→ 引出 **網頁爬取 / Browser 自動化** 概念
- 「為什麼可以自動操作瀏覽器？」→ 引出 **Browser Agent** 概念
- 「AI 怎麼分析正面負面？」→ 引出 **情緒分析** 概念
- 「報告怎麼自動產生的？」→ 引出 **模板化輸出 / 自動化報表** 概念

#### 對應知識模組

| 概念 | 生活化比喻 | 技術名稱 | 在 Antigravity 中怎麼做 |
|------|-----------|---------|----------------------|
| 自動看網頁 | 「派偵探去觀察對手」 | Browser Automation | Browser Sub-Agent |
| 為什麼能操作瀏覽器 | 「遙控器可以遙控電視」 | Headless Browser / DOM 操作 | Antigravity 內建 Browser 工具 |
| 正負面判斷 | 「看評論是誇還是罵」 | Sentiment Analysis | AI API Call |
| 自動產報告 | 「助理幫你整理成簡報」 | Template Rendering | Skill + 結構化輸出 |

**重點講解：為什麼可以用 Browser？**

```
你平常怎麼看競品粉專？
  → 打開 Chrome → 輸入網址 → 滑鼠滾動 → 眼睛看內容 → 腦袋記下來

程式怎麼做？
  → 打開虛擬 Chrome → 程式輸入網址 → 程式滾動頁面 → 程式讀取文字 → AI 分析內容

兩者做的事情完全一樣，只是一個是你手動，一個是程式代勞。
Browser 自動化 = 你僱了一個不會累的實習生，24 小時幫你看網頁。
```

#### 最小實作版
1. 用 Antigravity 的 Browser 工具打開一個公開粉專
2. 抓取最新 3 篇貼文內容
3. AI 分析每篇的主題和情緒
4. 產出簡單的文字報告

### 案例五：「智能排程助手」— 根據歷史數據建議最佳發文時間

#### 展示成品
輸入：「分析我們粉專過去 30 天的貼文，告訴我什麼時候發文互動最好」

輸出：
> 📈 發文時間分析報告
> - 最佳時段：週二、四 晚上 8-9 點（平均互動率 4.5%）
> - 次佳時段：週日 上午 10-11 點（平均互動率 3.8%）
> - 避免時段：週一 上午（互動率僅 1.2%）
> - 建議：影片型內容在週末表現最好，圖文在週間較優

#### 學員分組討論（引導問題）
- 「資料從 Google Sheets 讀的？還是直接從社群平台抓的？」→ 引出 **MCP（外部服務連接）** 概念
- 「MCP 是什麼？為什麼 AI 能讀我的 Google Sheets？」→ 引出 **MCP 原理** 講解
- 「AI 怎麼分析數據的？」→ 引出 **資料分析 + AI 推論** 概念
- 「能不能直接幫我排好貼文？」→ 引出 **Tool Calling / 工具呼叫** 概念

#### 對應知識模組

| 概念 | 生活化比喻 | 技術名稱 |
|------|-----------|---------|
| MCP 連接 | 「給助理一把鑰匙，讓他能進你的辦公室看資料」 | Model Context Protocol |
| 讀取外部資料 | 「翻出過去的業績表」 | Google Sheets API via MCP |
| AI 分析 | 「請顧問看數據給建議」 | Data Analysis + LLM |
| 執行動作 | 「不只給建議，還幫你做」 | Tool Calling / Function Calling |

**重點講解：MCP 到底是什麼？**

```
想像你有一個超聰明的助理（AI），但他被關在一個房間裡：
- 他很會分析、很會寫東西
- 但他看不到你的 Email、你的試算表、你的行事曆

MCP 就像是「幫這個房間裝上窗戶和門」：
- Google Sheets MCP → 他可以看到你的試算表了
- Gmail MCP → 他可以讀你的信了
- Calendar MCP → 他可以看你的行事曆了
- Notion MCP → 他可以讀寫你的 Notion 了

每一個 MCP Server = 一道「門」，連接到一個外部服務。
AI + MCP = 一個能看到你所有工具的智慧助理。
```

#### 最小實作版
1. 準備一份 Google Sheets（模擬的貼文數據）
2. 透過 MCP 連接 Google Sheets
3. AI 讀取數據並分析
4. 產出建議報告

## 五、核心概念教學卡片

### 概念 1：API — 程式之間的「共同語言」

**生活比喻**：
> 你去餐廳吃飯，你不會自己跑進廚房炒菜。你跟服務生說「我要一份牛排」，服務生去廚房傳達，廚房做好後服務生端給你。
> - 你 = 你的程式
> - 服務生 = API
> - 廚房 = 另一個服務（LINE、OpenAI、Google Sheets...）
> - 菜單 = API 文件（告訴你可以點什麼）

**為什麼學員需要知道**：
幾乎所有自動化都靠 API 串接。理解 API 就能理解「為什麼不同工具可以互相講話」。

**在 Antigravity 中的體現**：
當你跟 AI 說「幫我發一則 LINE 訊息」，AI 實際上就是在呼叫 LINE 的 API。

---

### 概念 2：Webhook — 「有事發生就通知我」

**生活比喻**：
> - **傳統方式（輪詢）**：你每 5 分鐘打一通電話問「我的包裹到了嗎？」
> - **Webhook**：你跟物流說「包裹到了就打給我」，你不用一直問。

**為什麼學員需要知道**：
聊天機器人、即時通知、自動化觸發，全部靠 Webhook。

---

### 概念 3：MCP — 讓 AI 有「鑰匙」存取你的工具

**生活比喻**：
> AI 很聰明但被關在房間裡。MCP 就是給它「不同房間的鑰匙」：
> - Google Sheets 的鑰匙 → 可以看你的表格
> - Gmail 的鑰匙 → 可以看你的信
> - Notion 的鑰匙 → 可以看你的筆記

**為什麼學員需要知道**：
這是 Antigravity IDE 最強大的功能之一。理解 MCP 就能理解「為什麼 AI 可以幫你操作 Google Sheets、讀你的 Email」。

---

### 概念 4：Browser 自動化 — 讓程式「像人一樣操作瀏覽器」

**生活比喻**：
> 你平常怎麼上網查資料？打開 Chrome → 輸入網址 → 點擊 → 複製文字。
> Browser 自動化就是讓程式做完全一樣的事，只是速度快 100 倍，而且不會累。

**為什麼學員需要知道**：
很多社群平台沒有開放 API，但透過 Browser 自動化，依然可以抓取公開資料、自動填表、截圖等。

**程式能做到什麼程度？**
- ✅ 自動打開網頁、滾動、點擊
- ✅ 自動讀取頁面上的文字
- ✅ 自動截圖
- ✅ 自動填寫表單
- ❌ 不能破解密碼或繞過安全機制
- ❌ 不建議用於違反平台規則的操作

---

### 概念 5：Skill — AI 的「工作說明書」

**生活比喻**：
> 你僱了一個新員工（AI），他什麼都能做，但需要你告訴他「做什麼、怎麼做」。
> Skill 就是你寫給他的「工作說明書」：
> - 工作名稱：每日產業新聞彙整
> - 什麼時候做：每天早上 8 點
> - 做什麼：去這 3 個網站抓新聞 → 寫摘要 → 發到 LINE
> - 注意事項：用繁體中文、不超過 200 字

**為什麼學員需要知道**：
Skill 是 Antigravity IDE 的核心。學會寫 Skill = 學會「指揮 AI 做事」。

**Skill 的結構其實就是三件事**：
1. 什麼時候做？（觸發條件）
2. 做什麼？（具體步驟）
3. 做完怎麼處理？（輸出 / 通知）

## 七、「與 AI 對話」技巧教學

### 學員常見的無效對話 vs 有效對話

| 無效 ❌ | 有效 ✅ |
|---------|---------|
| 「幫我做一個社群自動化」 | 「幫我做一個每天早上 8 點自動去 TechCrunch 抓最新 3 篇 AI 新聞，用繁中寫 50 字摘要，發到我的 LINE 群組」 |
| 「這個 code 有 error」 | 「我執行後出現這個錯誤訊息：[貼上錯誤]，我期望的結果是 [描述]，請幫我修正」 |
| 「加一個功能」 | 「在目前的晨報機器人基礎上，加一個功能：如果新聞提到我們的品牌名稱，就額外發一則通知給我」 |

### 對話三步驟（教學員的溝通框架）

```
第一步：說清楚「我要什麼成果」
  → 「我希望每天自動收到一份競品分析報告」

第二步：說清楚「輸入是什麼、輸出到哪裡」
  → 「從這 3 個競品的 Facebook 粉專抓資料，分析後發到我的 LINE」

第三步：說清楚「有什麼限制或要求」
  → 「用繁體中文、重點不超過 5 條、如果有負面評論要特別標註」
```

### 看不懂 Code 怎麼辦？— 教學員「問對問題」

不需要看懂 code，只需要學會問：
- 「這段程式在做什麼？請用非技術人員能理解的方式解釋」
- 「這段如果失敗了會怎樣？」
- 「我想修改 [某個行為]，應該改哪裡？」
- 「幫我加上中文註解，讓我看懂每一步在做什麼」

## 九、「程式到底能做到什麼」— 給非技術背景學員的認知校準

### 程式能做的事 ✅

| 類別 | 舉例 |
|------|------|
| **重複性工作** | 每天發相同格式的報告、定時抓資料 |
| **大量處理** | 一次分析 1000 篇留言的情緒 |
| **跨平台串接** | 收到 LINE 訊息 → 記到 Google Sheets → 通知 Slack |
| **24 小時運作** | 半夜收到客訴也能自動回覆 |
| **操作瀏覽器** | 自動打開網頁、截圖、抓資料 |
| **讀寫各種工具** | 透過 MCP 存取 Google Sheets、Notion、Gmail 等 |

### 程式不太能做的事 ❌

| 類別 | 說明 |
|------|------|
| **品牌創意判斷** | AI 可以產出文案，但最終品牌調性需要人決定 |
| **複雜人際互動** | 真正需要同理心的客訴處理，還是要真人 |
| **100% 準確率** | AI 會犯錯，重要決策需要人類審核 |
| **繞過安全機制** | 不能破解密碼、繞過驗證碼 |

### 正確的心態 🎯

```
自動化不是「取代你」，而是「幫你處理無聊的部分」。

你的價值在於：
- 決定「要自動化什麼」（策略）
- 判斷「AI 產出的東西好不好」（品質控制）
- 處理「機器人搞不定的事」（創意與同理心）

AI 是你的超級實習生，不是你的替代品。
```
